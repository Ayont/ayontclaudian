# 4.0.1 Design: Provider Stability, Renovate Cleanup & Universal Multi-Agent

## Goal

Make every supported provider stable and up-to-date, merge all open Renovate dependency PRs cleanly, and expose the existing multi-agent mission engine as a per-provider option with provider-branded UI colors.

## Scope

### 1. Renovate Dependency Cleanup

- Bump all open Renovate npm updates to their target versions.
- Resolve the `typescript-eslint` / `@eslint/js` peer-dependency conflict.
- Update Node.js runtime pin to `24.17.0` and GitHub Actions to `actions/checkout@v7` + `actions/setup-node@v6`.
- Verify typecheck, lint, tests, and build remain green.

### 2. Provider Audit & Update

- Audit each registered provider (claude, codex, kimi, grok, vibe/mistral, opencode, antigravity, pi) for:
  - Latest CLI command / flag changes
  - Correct model catalogs and default models
  - Working install detection
  - Health-check compatibility
- Fix any obvious breakage (wrong binary names, stale flags, missing models).
- Add missing provider metadata: `multiAgent`, `brandColor`, `brandColorLight`.

### 3. Universal Multi-Agent Option

- Add `supportsMultiAgent: boolean` to `ProviderCapabilities`.
- Default all providers to `supportsMultiAgent: true` except where technically impossible.
- Gate the "Multi-Agent Mission" command in the chat UI / dashboard on this capability.
- Allow per-provider override via settings if a user wants to disable it.

### 4. Provider-Branded Multi-Agent UI

- Extend `ProviderRegistration` with `brandColor: string` (hex) and optional `brandColorLight: string`.
- Use the active provider's brand color for:
  - Multi-Agent modal header / accent
  - Mission timeline / event log accent
  - Specialist agent badges when running under that provider
- Provide sensible fallbacks when no brand color is defined.

## Architecture

- Provider metadata lives in `src/core/providers/types.ts` and each provider's `registration.ts`.
- A new `MultiAgentAvailabilityService` reads the active provider's capabilities and decides whether to render multi-agent UI affordances.
- Brand colors are CSS variables injected into the dashboard/modal root based on the active provider.
- The existing `MultiAgentService` and `MissionStateStorage` from 4.0.0 are reused unchanged; only UI gating and theming change.

## Testing

- Add unit tests for `MultiAgentAvailabilityService`.
- Add tests verifying every provider registration declares `supportsMultiAgent` and a valid `brandColor`.
- Run full test suite, linter, typecheck, and production build.

## Out of Scope

- New multi-agent algorithms or mission types.
- Message search / prompt library (remains backlog for 4.1.0).
- Provider runtime rewrites beyond flag/model fixes.
