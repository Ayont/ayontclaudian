# Claudian Obsidian Plugin 5.0.0 Release Design

## Goal

Release version **5.0.0** of the Claudian Obsidian plugin by integrating the four already-implemented streams (S1–S4) into `main`, ensuring the codebase passes lint, type-check, and the full test suite, then bumping the version and producing release artifacts.

## Scope

### In Scope (5.0.0)

| Stream | Summary | Worktree Branch |
|---|---|---|
| S1 | Universal soft-steer for all providers | `5.0.0-s1-steer` |
| S2 | Multi-agent team engine (20 agents, cross-provider, inline `/team`) | `5.0.0-s2-multiagent` |
| S3 | Dashboard 2.0 (interactive browser modals, token-usage chart) | `5.0.0-s3-dashboard` |
| S4 | Prompt templates (`/template`) + vault health commands (`/vault-health`) | `5.0.0-s4-templates` |

### Out of Scope (moved to 5.1.0)

- `5.0.0-s5-context`
- `5.0.0-s6-routing`
- `5.0.0-s7-branching`
- `5.0.0-s8-diff`

These worktrees currently contain only the base commit and have no defined features, specs, or implementation. They will go through their own design → plan → implement cycles for a future 5.1.0 release.

## Current State

- `main` is at `bb498aa`, 7 commits ahead of `ayont/main`.
- All four S1–S4 worktrees share the same merge-base with `main`: `ac98258`.
- `main` already contains large portions of S2/S3 code; the worktree branches carry the remaining incremental changes plus removals of obsolete test files.
- S4 has minor lint issues (unused imports/variables) that must be fixed before merge.

## Approach

### Merge Strategy

Use **rebase + merge** (Approach A):

1. Rebase `5.0.0-s1-steer` onto `main`, resolve conflicts, merge into `main`.
2. Rebase `5.0.0-s2-multiagent` onto the updated `main`, resolve conflicts, merge into `main`.
3. Rebase `5.0.0-s3-dashboard` onto the updated `main`, resolve conflicts, merge into `main`.
4. Rebase `5.0.0-s4-templates` onto the updated `main`, fix lint issues, resolve conflicts, merge into `main`.

This keeps history linear and forces per-stream conflict resolution.

### Quality Gates

After each rebase/merge and once at the end:

- `npm run lint` — 0 errors.
- `npx tsc --noEmit` — 0 type errors.
- `npm test` — all tests pass.
- `npm run build` — produces `main.js` and `styles.css`.

### Version Bump

After all streams are merged and green:

- `manifest.json`: `5.0.0`
- `package.json`: `5.0.0`
- `versions.json`: add `"5.0.0": "0.15.0"` (keep existing entries).

### Release Artifacts

- `main.js`
- `styles.css`
- `manifest.json`
- Updated release notes (`docs/release-notes-5.0.0.md`).

## Error Handling

- If a stream fails lint/tests after rebase, fix it in its worktree before merging.
- If a conflict is unresolvable without design changes, escalate to the user with the specific files.
- If the known flaky `TitleGenerationService.test.ts` fails, retry the suite; do not block the release for an unrelated pre-existing flake.

## Testing Strategy

- Run the full Jest suite after each stream merge.
- Run a smoke build after each stream merge.
- Final verification: lint, type-check, full test suite, build, and manual manifest/version consistency check.

## Risks

| Risk | Mitigation |
|---|---|
| Merge conflicts in `src/main.ts` or `InputController.ts` | Resolve per-stream; use tests to verify behavior |
| S4 lint failures block merge | Fix unused imports/variables before merging S4 |
| Flaky `TitleGenerationService.test.ts` | Retry; if it persists, document it and do not gate 5.0.0 |
| S2/S3 partially already in `main` | Review diff carefully to avoid duplicate code |

## Success Criteria

- `main` contains all S1–S4 changes.
- `npm run lint`, `npx tsc --noEmit`, `npm test`, and `npm run build` all pass.
- Version is `5.0.0` in `manifest.json`, `package.json`, and `versions.json`.
- Release notes `docs/release-notes-5.0.0.md` exist and describe S1–S4.
